# post-type: tip

tips / コツ / before-after の共有。**数字フック** や **小さな気づき** が
効く type。

## いつ使うか

- 短時間でできるテクニックを共有したいとき
- before / after で改善が示せる事例
- 「最近こうやってる」系の運用 tip
- 設定一発で何かが楽になる系（"〜を有効にしたら〜になった"）

## 典型構成

### フック（1 行目）

候補:

- **数字 before/after**: 「3 時間かかってた作業が 30 分になった」
- **逆説**: 「〜と思ってたけど違った」
- **具体名**: 「`grep -P` の代わりに `ripgrep` を使うと〜」

### 本文（中央）

before / after を **対比可能な形** で書く。あるいは状況 → 学び。

### CTA

- リンクは **任意**（tip 自体が完結していればなくても OK）
- リンクを付けるなら、深掘り資料（ブログ・Zenn 記事 等）

## 推奨文字数

- 全体: 140-200 字（短めが効く）

## 典型添付パターン

| 添付 | 文面の役割 |
|---|---|
| なし | 文面のみで完結 |
| 1 枚（before / after が見える図）| 文面は数字や教訓に集中 |
| 2 枚（before + after）| 各画像の説明は文面では繰り返さない |
| 動画 | 「短い動きで違いが見える」場合に効く |

## 例文

### 例 1: understated × oss-dev × 添付なし

> Claude Code のスキルで persona をファイル別に切り出すと、後から追加する
> ときに SKILL.md を触らずに済むので運用がラクでした。
> 進化させやすい構造、推せます。

### 例 2: understated × oss-dev × 添付 2 枚（before/after）

> Claude Code のスキル設計、最初は SKILL.md に persona を全部書いていま
> したが、ファイル別に切り出してから「真面目」「カジュアル」を増やすのが
> 一気に楽になりました。
> 拡張容易性は最初から効く。

→ 添付 2 枚: 1 枚目 = SKILL.md 全部詰めた版、2 枚目 = 切り出した version。
文面では各構造を詳細描写しない（画像で見える前提）。

### 例 3: casual × general × 添付なし

> AI ツール導入で「これ何に使うんだ？」って一瞬でも止まったとき、
> その「何に使うか」を一緒に考えてくれるツールがあると一気に楽
> なんですよね 🙂

→ 共感型の tip。具体ツール名を出さなくても成立する場合は、抽象でも OK。

## NG 構成

- 「知らないと損」「全人類が使うべき」（煽り）
- 数字が大袈裟（"100 倍速くなりました" を実証なしで)
- 「結論から言うと」で始める
- 詳細詰め込みすぎ（→ 別途記事に逃がす）

## tip type で迷ったら

- 数字 / 比較 / 具体名 のいずれかが冒頭にあるか
- 1 ツイート単独で完結しているか（リンクなしでも価値が伝わる）
