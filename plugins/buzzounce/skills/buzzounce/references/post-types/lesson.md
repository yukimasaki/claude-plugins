# post-type: lesson

失敗談 / 学び / 気づきの共有。**軽い自虐** が活きる type。

## いつ使うか

- ハマったポイントから得た教訓を共有したいとき
- 「〜と思ってたけど違った」系の認識アップデート
- ツール / 仕様の落とし穴の共有
- 検証 / 実験の結果が想定と違ったとき

## 典型構成

### フック（1 行目）

候補:

- **「Y で詰まった」**: 「release-please の skip 機能で詰まりました」
- **「Z だと思ったら違った」**: 「`<video>` タグで動画埋め込めると思って
  たら、GitHub では再生されませんでした」
- **「X を試したら Y になった」**: 「retina で render したら SPI が 5MB
  超えて却下された」

### 本文（中央）

**状況 → つまずき → 教訓** の流れが王道。

- 状況: 何をしようとしていたか
- つまずき: 何が起きたか
- 教訓: 結局どうしたか / 何が学びだったか

教訓を **1 文で言い切れる** か確認。長い解説は外部リンクへ逃がす。

### CTA / リンク

- 詳細記事 / 経緯メモへのリンクが効く
- なくても成立する（教訓が完結していれば）

## 推奨文字数

- 全体: 140-280 字
- 中盤の状況描写は冗長になりがちなので、徹底して圧縮する

## 典型添付パターン

| 添付 | 文面の役割 |
|---|---|
| なし | 文面のみで成立 |
| 1 枚（エラー画面 / 失敗の証拠）| 教訓側に集中 |
| 2 枚（before / after） | 教訓を 1 点に絞る |

## 例文

### 例 1: understated × oss-dev × 添付なし

> release-please の `Release-As` で release を skip しようとしたら、同じ
> version の Release PR が再生成され続けて詰まりました。
> 公式に skip 機能はないようなので、新しい version を一回切る方が早かった
> です。
>
> 経緯メモ: https://example.com/your-project/your-repo/blob/...

### 例 2: understated × oss-dev × 添付 1 枚（証拠）

> README に `<video>` タグで動画を埋め込もうとしたら、GitHub WebUI では
> 再生されないことがあるようでした（attribute サニタイズが原因かも）。
> 結局 raw URL を Markdown で貼る形 + 静止画 PNG の併用に切り替えて
> 落ち着いています。

### 例 3: casual × general × 添付なし

> AI ツールでヒーロー画像作って、retina (2 倍) で書き出したら GitHub
> Social Preview の上限 5MB を超えて却下されました…。
> 結局 1 倍で書き出し直すのが正解でした 🥲

## NG 構成

- 「やらかした」「終わった」を強調しすぎる（自己卑下が過ぎると重い）
- 失敗を笑いで済ませるが教訓がない（→ tip にすべき）
- 他人のせいにするニュアンス（"このツールの仕様がおかしい"）
- 詳細を全部書こうとして長くなる（→ 経緯メモを外部に逃がす）

## lesson type で迷ったら

- **状況・つまずき・教訓** の 3 要素がそれぞれ 1-2 文に圧縮できているか
- 教訓が読者にも適用可能な形で抽出されているか
- 失敗の直接的描写ではなく、「学び」が中心になっているか

軽い自虐は understated / casual で活きるが、formal では使わない（formal は
失敗共有でも事実列挙にとどめる）。
