# post-type: dogfood

自作のツールで自作のための素材を作った系の **自己参照ストーリー**。
「ツール自身が自分のマーケ素材を作る」という構造的に強いフックを使う。

## いつ使うか

- 自分のプラグイン / ツールで、そのプラグインの README hero / SPI / 告知
  画像を生成したとき
- 自分のスキルに自分の告知ツイートを書かせたとき
- 自作ライブラリを別プロダクトで実戦投入したとき
- ツール自体の用途・有用性を、ツール自体の出力で証明できる場合

## 典型構成

### フック（1 行目）

候補:

- **「X を自分の Y に作らせた」**: 「README の hero 画像、自分のプラグイン
  に作らせました」
- **「自分の Y を X が作った」**: 「告知ツイートを buzzounce が書きました」
- **「X が X を Y する」**: 「buzzounce で buzzounce の告知文を書く」

ポイントは **自己参照のループ** を 1 行で示すこと。読者の頭に「あ、面白い」
が立つ瞬間を作る。

### 本文（中央）

**ツール自体の説明** を簡潔に + **自己参照によって何が良かったか / 何が
分かったか** を 1 文。

「これがそのまま製品の使い方デモになっている」みたいな構造を意識すると、
読者にとっても価値が伝わる。

### CTA / リンク

- リンク先には「制作プロセス」「全プロンプトログ」など、ツール自体の動きが
  追体験できるものを置く
- 単なる README リンクより、「making-of」「session log」「sample/」が
  あれば優先

## 推奨文字数

- 全体: 140-280 字
- 自己参照の構造で 1 つフックが効くので、短めでも成立しやすい

## 典型添付パターン

| 添付 | 文面の役割 |
|---|---|
| 1 枚（自作で生成した素材） | process / why のストーリー |
| 2-4 枚（before / 生成中 / 完成 / 別バリ） | プロセスの 1 場面に絞る |
| 動画（生成過程のスクキャス）| 「最後の納品まで見てほしい」と誘導 |

## 例文

### 例 1: understated × oss-dev × 添付 1 枚

> README の hero 画像、自分のプラグインに作らせました。
> design-director — Claude Code 上で対話するだけで UI プロトタイプを並列
> 生成して、React JSX として納品まで進む個人 OSS です。
> 制作プロセス（プロンプト全公開）はリポへ。
>
> https://example.com/your-org/your-repo

### 例 2: understated × oss-dev × 添付なし

> このツイート文、buzzounce に書かせました。
> persona と mode を選ぶだけで日本語の告知ツイートが対話で組み立てられる
> スキルなので、自分の告知も自分のスキルに書かせるとちょうど良かった
> です。
>
> https://example.com/your-org/your-repo

→ 文面の冒頭で「これが buzzounce の出力です」と明示。実演兼ねた告知。

### 例 3: casual × general × 添付 1 枚

> 自分のプラグインに自分の告知ツイートを書かせてみたんですが、これが結構
> 効くんですよ ✨
> 「画像 1 枚目に何があるか」まで聞いてくれるので、文面と画像が重複
> しないように調整してくれます。
>
> https://example.com/your-org/your-repo

## NG 構成

- 自己参照を **遠回しに** ほのめかす（読者が自己参照に気づかない）
  → 1 行目で明示する方が効く
- 「ぜひ皆さんも使ってみて、自作のツールで〜」と過剰に誘導する
- 「すごいでしょ？」感が滲む（"これ作ったの私です" 系）
- ツールの説明だけで、自己参照ストーリーが立っていない（→ それなら
  release type を選ぶべき）

## dogfood type で迷ったら

- フックを声に出して読み「あ、それ面白い」と一瞬で思えるか
- 思えなければ release / show-and-tell / lesson のいずれかに切り替え
- self-check で「自己参照のループ」が 1 行で示せているかを確認
