# self-check — 投稿前チェックリスト

候補を生成したあと、ユーザーに確定案を渡す **前に** silently 走らせる
7 項目チェック。1 つでも引っかかったら step (3) 選択深掘りに戻り、修正
してから再チェック。

## 7 項目

### 1. ターゲット読者が明確か

- 「誰に向けた投稿か」が文面から推測できるか
- mode（oss-dev / general）と post-type が一致しているか
- 例: oss-dev mode で「副業始めたい人へ」になっていないか

### 2. 具体的な事実（URL / 数値 / 名前）が含まれているか

- 「使ってみたら良かった」だけで終わっていないか
- リリース告知なら version 番号 or 主要機能名が入っているか（ただし
  rotting info に注意 → `anti-slop.md`）
- tip 系なら before/after の数値があるか
- show-and-tell なら何を作ったかの具体名

### 3. リンクを踏む動機があるか

- リンク先で何が見られるかが想起できるか
- 「詳細はこちら」だけでは弱い、「制作プロセスの全公開ログ」「ベンチマーク
  結果」「ライブデモ」など、踏む理由を一行で示す
- 例外: 動画 / 画像で十分伝わっていて、リンクは「興味あれば」程度の
  オプションに留めたい場合は OK

### 4. 過剰自己宣伝になっていないか

- 「！」が 2 個以上ある → 削る
- 「ぜひ」「必ず」「絶対」が入っている → 控えめな表現に
- リプライ要求（保存推奨・RT お願いします等）が入っている → 削除
- 自己評価語（自信作・渾身の・神）が入っている → 削除

### 5. 選択した persona の声に整合しているか

- `personas/{選択}.md` の「文末」「一人称」「許される表現」「NG」と
  矛盾していないか
- 例: `understated` 選択時に「最強です！」が入っていないか
- 例: `formal` 選択時に体言止めが多用されていないか

### 6. anti-slop の NG リストに該当していないか

- `anti-slop.md` の 7 カテゴリそれぞれを見直す
- 特に「7. 嫌な印象を与える表現全般」は読み手視点で再確認

### 7. 添付素材の内容と文面が重複していない / 補完関係になっているか（添付ありの場合）

- 1 枚目に既に映っているもの（hero / UI / ロゴ等）を文面でも詳細に説明
  していないか
- 動画で動きが見せられる場合、文面で動きを再描写していないか
- 添付なしの場合は「文面単独で全部伝える」が成立しているか

詳しい判断基準は `attachments.md`。

## チェック結果の伝え方

silently に実行し、ユーザーには以下の形式で報告:

```
self-check 結果:
✅ #1 ターゲット明確 (oss-dev / dev コミュニティ向け)
✅ #2 具体的事実 (v0.2.0、機能名 X が含まれている)
✅ #3 リンク踏む動機 (制作プロセスログを開示)
⚠️  #4 過剰自己宣伝 (「ぜひ」を含む) → 修正提案: 削除
✅ #5 persona 整合 (understated)
✅ #6 anti-slop 非該当
✅ #7 添付との補完性 (1 枚目=hero、文面はストーリー側に寄っている)
```

⚠️ がある場合は、ユーザーに「この修正を当てますか？」と確認してから
反映する。

## 注意

- self-check は **品質ゲート** であって、ユーザーの意図を勝手に書き換える
  装置ではない
- ユーザーが「これでいい」と言えば、項目 NG のままでも納品する
  （loud で警告は残す）
- 0.1.0 では self-check 結果のメモリ保存はしない（毎回 fresh）
