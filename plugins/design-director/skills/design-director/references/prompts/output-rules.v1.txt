# Output rules

## Artifact wrapper

Every design must be delivered inside exactly one artifact tag:

```
<artifact identifier="design-1" type="html" title="Concise title here">
<!doctype html>
<html lang="en">
  ...
</html>
</artifact>
```

- `identifier`: slug form, e.g. `design-1`, `landing-hero`, `settings-screen`
- `type`: always `html` for HTML prototypes
- `title`: 3-6 words, describes what the artifact is (not what you did)

No second artifact tag. No Markdown fences. No `<!--comments-->` outside the `<html>`.

## File constraints

- **Maximum 1000 lines** of HTML (including inline style and script). If the design would exceed this, simplify ‚Äî omit repetitive cards, reduce copy, consolidate sections.
- Self-contained: no `<link rel="stylesheet">`, no `<script src="‚Ä¶">` to your own files.
- Permitted external resources (tightly scoped ‚Äî same trust policy as Claude Artifacts):
  - **CSS**:
    - Tailwind CDN: `<script src="https://cdn.tailwindcss.com"></script>`
    - Google Fonts: `<link rel="preconnect">` + `<link rel="stylesheet">` from `fonts.googleapis.com` / `fonts.gstatic.com`
  - **JS libraries** ‚Äî `cdnjs.cloudflare.com` whitelist only. Pin an exact version. Format: `https://cdnjs.cloudflare.com/ajax/libs/<lib>/<exact-version>/<file>.min.js`. Approved libraries:
    - `recharts` ‚Äî data viz (preferred for dashboards)
    - `Chart.js` ‚Äî alternative charting (note: cdnjs slug is capitalized)
    - `d3` ‚Äî low-level visualization
    - `three.js` ‚Äî 3D
    - `lodash.js` ‚Äî utilities (cdnjs slug includes the `.js`)
    - `PapaParse` ‚Äî CSV parsing (note: cdnjs slug is CamelCase)
- **Forbidden**:
  - Arbitrary `fetch()` / `XMLHttpRequest` to external APIs ‚Äî all data must be inline.
  - Scripts from any host other than `cdnjs.cloudflare.com` (no `esm.sh`, `jsdelivr`, `unpkg` ‚Äî too open, no version verification).
  - Hotlinked photos from any host (`placeholder.com`, `unsplash.com`, `picsum.photos`, etc.).
- All other assets must be inline: SVG icons, CSS gradients, data URIs for tiny images.

## CSS custom properties (required)

Declare every load-bearing visual value as a CSS custom property on `:root`:

```css
:root {
  --color-bg:       #f8f5f0;
  --color-surface:  #ffffff;
  --color-text:     #1a1a1a;
  --color-muted:    #6b6b6b;
  --color-accent:   oklch(62% 0.22 265);
  --color-accent-2: oklch(72% 0.18 40);
  --radius-base:    0.5rem;
  --radius-lg:      1rem;
  --font-sans:      'Syne', system-ui, sans-serif;
  --font-mono:      'JetBrains Mono', monospace;
  --space-unit:     1rem;
}
```

Reference these in Tailwind's arbitrary-value syntax: `bg-[var(--color-accent)]`, `rounded-[var(--radius-base)]`. Never hard-code hex or pixel values in Tailwind classes when a variable covers the same slot.

## Structural rules

1. Semantic landmarks: `<header>`, `<main>`, `<section>`, `<article>`, `<nav>`, `<footer>` ‚Äî one each where appropriate.
2. Heading hierarchy: one `<h1>`, then `<h2>` per section, `<h3>` for sub-items. Never skip levels.
3. Interactive elements: `<button>` for actions, `<a href="#">` for navigation. Never `<div onclick>`.
4. Images: no hotlinked photos. Use inline SVG compositions or CSS gradient placeholders.
5. Alt text: every `<img>` has a non-empty `alt`. Decorative SVGs get `aria-hidden="true"`.
6. No `<table>` for layout; use CSS grid or flex.
7. Responsive: mobile-first breakpoints using Tailwind's `sm:`, `md:`, `lg:` prefixes.
8. Motion: CSS `transition` / `animation` only ‚Äî no JS animation loops (no `requestAnimationFrame`, no recursive `setTimeout` for visuals). Keep it under 300 ms unless the effect is intentional and earns its cost. The single permitted exception is the dashboard live-clock `setInterval(updateClock, 1000)` documented in the craft directives.

## Content rules

- No lorem ipsum. Write copy specific to the domain the user described.
- No placeholder names like "John Doe" or "Company Name" ‚Äî invent plausible, diverse names.
- Numbers and dates must be realistic (not "100%" everywhere, not "Jan 1, 2020").
- Icons: inline SVG only; use simple, recognizable symbols (no brand logos without explicit request).

## Rotting information in long-lived artifacts

Marketing material, README headers, hero images, social-preview images, pitch decks ‚Äî anything intended to live for months or years ‚Äî must not embed values that decay over time. These rot fast and force the author to re-render the artifact for every release.

Forbidden in long-lived artifacts:

- Concrete version numbers (`v0.1.1`, `Version 2.3`). Use a badge that reads from the source of truth instead (npm, GitHub release, package manifest), or omit entirely.
- Calendar dates and years (`¬© 2026`, `Updated March 2026`). Footer years should be generated from `new Date().getFullYear()` *outside* the artifact, or omitted.
- Ordinal claims that drift (`our first plugin`, `the only tool that ...`, `Á¨¨‰∏ÄÂºæ`). Replace with neutral framing ‚Äî describe what the thing *is*, not its position in a timeline.
- Quantities that change with the codebase (`9 subcommands`, `12 themes`, `24 components`). Either pull live from the source at render time, or describe the category without counting.
- Roadmap references (`coming Q2`, `available next month`). These age the artifact the moment the date passes.

When the user asks for a hero, README graphic, or any artifact destined for a public surface, run a "rot check" before delivery: scan the copy for versions, years, ordinals, and counted nouns, and either neutralize them or surface them through a badge / dynamic source. If something *must* be pinned (a launch announcement tied to a specific version), call it out explicitly so the user knows the artifact has a planned shelf life.
